PhonePe Interview Experience — Full-Time SDE Role 2023

As much as coding interviews have trended away from “Leetcode-style,” they are still a very unique skillset that is far from your day-to-day job as a software engineer. Think about coding interviews like a certification exam. It might not be what 90% of your job is like, but you need to master the fundamentals to get a job. The Spectrum of Coding Interviews Coding interviews come in a variety of flavors: 2–3 Leetcode-style problems, usually these are distinct and have nothing in common. Practical problems with 2–3 parts, usually these build on one-another. They are self-contained and focus on data structures and algorithms. Debugging interviews—Given a codebase and some bugs, figure out where the bug is and fix it. Coding Assignment—Implement some sort of API or library that does something, usually having to make some sort of remote call. For example, calling an API, a DB, or just another socket. Coding interviews seemed to have trended away from Leetcode-style towards practical. In 2022, even though Leetcode-style is becoming less favorable its still a great way to practice. You need to practice coding these types of problems and you simply can’t get that practice at work. Grind Leetcode This is the quickest way to get better at data structures and algorithms. Problems are well-scoped. Solutions are widely available. And you have a lot to practice. Here are some great starting resources: Blind75 — This is the basis for the following lists https://neetcode.io/ https://seanprashad.com/leetcode-patterns/ https://www.youtube.com/c/NeetCode Do a first pass of the listed problems. If you don’t immediately understand how to do the problem, it’s okay to look at the Discuss section and see how other people have solved it. Some problems may have impenetrable solutions. If you get stuck, try to find more solutions on Leetcode, YouTube, Medium, or Github. You might notice a pattern or find a different approach to the problem that comes easier to you. For example, I always preferred solving problems in recursively because it made more sense in my head. So I always sought out the recursive solutions when I was stuck. Try to diagram the solution. Diagramming and visualizing code can unstuck yourself. Try to implement a solution based off of what you read but without copying it line by line. At a later point, go back over the problems and try to do them from memory without looking at the solutions. If you find yourself banging your head over the same type of problem try doing a lot of different problems in that area. Leetcode organizes problems by tags. It’s really helpful to do other similar problems so that you can pick up a general pattern to apply to those same types of problems. Some of my the tags I spammed include: https://leetcode.com/tag/depth-first-search/ https://leetcode.com/tag/topological-sort/ https://leetcode.com/tag/monotonic-stack/ https://leetcode.com/tag/union-find/ https://leetcode.com/tag/divide-and-conquer/ Once you do similar types of problems enough, try to come up with some mental shortcuts and templates. How can you name variables similarly so that you don’t have to decide how to name them in the interview? Come up with templates for the most common algorithms—BFS and DFS. And understand the ways to transform them for your needs—Dijkstra’s, Backtracking. A non-exhaustive list of some algorithms and data structures to memorize BFS—Iterative and Recursive DFS—Iterative and Recursive Union-Find Tries Stacks—Parse and execute a simple arithmetic expression Know your language and ecosystem Coding your solution can be made much simpler if you know some useful libraries available in your language of choice. Interviewers commonly look for whether or not you have mastery of your language of choice. The best way to learn new tricks of your language is to look at other peoples’ solutions on Leetcode. Here are some common libraries I use in Python: Format Strings f-strings Objects and Tuples collections.namedtuple Memoization functools.cache Data Structures frozenset, defaultdict Binary Searching: bisect.bisect_left and bisect.bisect_right Min/Max Heap: heapq.heap, heapq.mergify Besides core data structures and algorithms, know how to interact with basic IO. Know how to make HTTP requests and read and write files. Web requests—In python know the requests library Read, write, and parse files—In python know open,split,json.loads, and json.dumps. An extra good-to-have (especially if your interviewer is themselves an expert at the language), know how to write idiomatic code in the language of your choice: How to set default values in function arguments or dictionaries How to write multi-line strings How to handle exceptional cases How to name your variables—Global, Instance, Snake, Camel How to write for-loops—Over an integer range and over collections How to specify data tags—privacy, const-ness How to assign multiple variables or unpack values Debugging Process If you have to debug your code or you’re given a debugging problem I strongly prefer littering your code with print statements over using a debugger. I always print all intermediate data structures until I isolate where the problem is happening. I don’t delete any print statements until I’ve found the bug. I recommend print statements over a debugger for three reasons: Iteration speed—You can quickly retry with new fixes or inputs Portability—You can do this in browser editors Context—You can see everything that’s happening all at once One caveat: A debugger can be helpful if there are ✨ magical ✨ functions. Most debugging interviews won’t have dynamically generated methods or decorators. But in this one case a debugger can give you more information about the execution path of your program.
